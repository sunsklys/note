# 项目介绍

## 项目概览

hooloo 是一个 智能自动售卖/咖啡机器人 的后端系统，包含 SaaS 管理平台、IoT 设备控制、ROS 机器人通信、工控机上位机等多个子系统。  
项目分为 6 大模块，包含 18 个独立入口。

## 二、worker/ — 异步任务消费者

1. worker/main.go — 异步任务处理器
   多模式异步任务处理器，每种模式解决不同的业务问题：

#### make — 单店铺订单制作处理器

一个进程只服务一个店铺，是该店铺完整的"订单制作大脑"。  
负责整个订单从付款到取餐的全流程闭环：

1. 接单：监听 Redis 队列，获取已付款的新订单
2. 生成制作指令：将订单拆解为 SKU 级别的制作指令（RISC 指令）
3. 发送指令：通过 MQTT 将制作指令下发给工控机
4. 接收结果：消费工控机返回的制作结果（通过 AMQP）
5. 取餐处理：处理取餐队列、取餐结果
6. 异常处理：订单超时未支付、异常状态恢复、机器人空闲检测
7. 交互处理：处理 ROS 交互指令、太空舱订单、控制面板操作
8. WebSocket 推送：将状态变化推送给前端
   启动时通过 -s 指定店铺 ID，一个进程只管一家店。如果有多家店铺需要同时制作，就要启动多个进程。

#### dis — AMQP 消息分发器

消息路由中间件，不直接处理业务，只做"分发"。  
工控机完成硬件操作后，通过 AMQP 消息队列返回结果。dis 模式消费这些 AMQP 消息，根据消息类型分发到对应店铺/机器的 Redis 队列中：

| 消息类型                       | 分发到                            |
| ------------------------------ | --------------------------------- |
| 属性检查结果                   | 机器属性处理队列                  |
| 业务监控数据                   | 机器业务监控队列                  |
| 制作指令结果监控               | 机器制作指令结果队列              |
| 交互事件                       | 店铺交互处理队列                  |
| 太空舱交互事件                 | 店铺太空舱交互处理队列            |
| 控制面板操作结果               | 控制面板结果队列                  |
| 故障恢复信息                   | 恢复日志队列                      |
| 咖啡机报警信息                 | 咖啡机信息队列                    |
| Galbot 状态错误                | Galbot 错误队列                   |
| 重做指令                       | 按制作指令结果处理（fallthrough） |
| 默认（含制作结果、取餐结果等） | 店铺制作结果队列                  |

dis 是"多对多"的路由层：多个工控机的消息 → 分发到多个店铺的处理队列。make 模式再从各自的队列中消费处理。

#### hub — 多店铺集中处理器

一个进程同时服务多个店铺，是 make 模式的多租户升级版。  
make 模式每个店铺需要独立进程，hub 模式将所有店铺的制作逻辑收敛到一个进程内，每个店铺对应一组独立的协程（ShopWorker），互不干扰。

- 支持通过命令行参数指定初始店铺列表
- 支持运行时动态增减店铺（收到 open-platform 指令时自动启停）
- 带重试机制的启动链路，防止并发重复启动
- 适合多店铺同时运营的场景，减少进程数量和资源开销

#### 三者关系

```plaintext
工控机硬件 ←→ MQTT ←→ [dis 分发器] → Redis队列 ← [make/hub 制作处理器] → 订单完成
                     AMQP消息路由          ↑                    订单制作全流程
                   (多对多分发)        队列消费              (单店或多店)
```

- dis = 消息路由层（纯分发，不处理业务）
- make = 单店铺制作处理器（一进程一店）
- hub = 多店铺制作处理器（一进程多店，make 的升级版）

## 四、iot/ — IoT 设备控制层

1. iot/main.go — IoT HTTP API 服务
   工控机本地 HTTP 服务，运行在工控机上，为设备端提供轻量 REST API。

- 健康检查：HEAD /health — 供云端探活，判断工控机是否在线
- 文件上传：POST /common/upload — 通用文件上传接口
- 静态文件服务：托管设备端页面（管理界面、配置页面等）
- 定位：与 iot/script 功能互补 — 本服务提供 HTTP API（被云端调用），iot/script 提供命令行工具箱（本地执行）

1. iot/socket/main.go — WebSocket 服务
   工控机 WebSocket 服务器，是设备端与前端屏幕（H5 点单屏、KDS 厨房显示屏）之间的实时通信桥梁。

#### WebSocket 端点

| 端点             | 业务用途                                 |
| ---------------- | ---------------------------------------- |
| /ws              | 主连接 — 推送业务事件 + 语音购物数据     |
| /ws100           | 备用连接（HL100 机型专用）               |
| /ws-machine-code | 读取机器码（用于设备注册/绑定）          |
| /tianqi          | 天奇系统订单交互（创建订单、查询、打印） |

#### 推送给屏幕的业务事件

通过 Redis 消费者获取云端数据，实时推送到前端屏幕：

| 事件 Action         | 业务含义                        | 数据来源       |
| ------------------- | ------------------------------- | -------------- |
| order_created       | 订单创建成功                    | Redis 队列消费 |
| order_status        | 订单状态变更（制作中/已完成等） | Redis 队列消费 |
| pos_pay_amount_sent | POS 机金额已发送                | Redis 队列消费 |
| payment_progress    | 支付进度更新                    | Redis 队列消费 |
| payment_complete    | 支付完成                        | Redis 队列消费 |
| payment_failed      | 支付失败                        | Redis 队列消费 |
| print_request       | 请求打印小票                    | Redis 队列消费 |
| SKU_STATE_CHANGED   | 商品上下架状态变更              | 业务数据消费   |
| QRCODE_VISITED      | 二维码被扫码访问                | 业务数据消费   |
| GOODS_CHANGED       | 商品信息变更                    | 业务数据消费   |
| BANNER_CHANGED      | 首页 Banner 变更                | 业务数据消费   |

#### KDS 厨房显示屏

- 订单制作进度推送：实时推送订单制作状态到 KDS 屏
- 扫码数据广播：消费 Redis 扫码队列，将扫码结果推送到 KDS 屏显示

#### 前端屏幕 → 服务端交互

- 创建订单：前端通过 /tianqi 端点发送 create_order 指令 → 调用 brain 后端创建订单
- 打印小票：前端发送 print_request 指令 → 触发打印
- 查询订单：前端发送 order_query 指令 → 查询订单状态
- 语音购物：消费 Redis Stream 中的语音购物结果，推送到前端显示

1. iot/script/main.go — 工控机核心服务
   工控机上最核心的服务，负责接收云端制作指令并驱动所有硬件完成饮品制作。提供三种工作模式：

#### sub mqtt risc start — MQTT 订阅核心（饮品制作引擎）

整个系统"从订单到出杯"的关键链路：

```plaintext
顾客下单 → brain 创建订单 → worker 生成 RISC 指令 → MQTT 下发
→ 工控机接收 → 按指令序列驱动硬件 → 出杯完成
```

MQTT 消息类型及对应硬件动作：

| RISC 指令类型                   | 触发的硬件动作              | 业务含义             |
| ------------------------------- | --------------------------- | -------------------- |
| pick（取杯/取料）               | PLC 从仓位取杯子/原料       | "准备一个杯子"       |
| abandon（弃杯）                 | 机械臂丢弃当前杯子          | "杯子异常，丢弃重做" |
| interaction（交互）             | ROS 交互指令                | "与机器人交互"       |
| interaction_space（太空舱交互） | 太空舱 ROS 交互指令         | "太空舱机器人交互"   |
| recover（恢复）                 | 机器人恢复安全位置          | "回到安全点"         |
| operate（控制面板）             | 运维面板操作                | "现场运维控制"       |
| cancel（取消制作）              | 中止当前制作流程            | "取消这个订单"       |
| redo（重做）                    | 重做某条指令                | "重试上一步"         |
| order_source（本地订单）        | LED 显示                    | "更新屏幕显示"       |
| socket_interaction              | 推送到点单屏                | "同步状态到前端"     |
| config（配置同步）              | 缓存机器配置                | "同步云端配置"       |
| ProxyG1（太空舱）               | G1 机器人 WebSocket 通知    | "太空舱指令"         |
| tianqi（天奇）                  | POS刷卡/移动支付/订单缓存   | "支付相关操作"       |
| 默认（制作指令）                | 保存到 Redis 队列，按序执行 | "排队制作"           |

硬件驱动覆盖：PLC、继电器、温湿度传感器、制冰机、咖啡机、冰淇淋机、麻薯机、银花机（珍珠机）、标签打印机、LED 屏、扫码枪、电磁锁、重量传感器、POS 机、嵌入式主板、杯托、液体泵、果酱机、灯带、机械臂  
按机型差异初始化：标准机型（V1）初始化 PLC/继电器/温湿度/咖啡机/机械臂/冰机/LED/锁/重量传感器；HL100/DeskA/DeskB（V4）额外包含嵌入式/印花机/冰淇淋/出杯机，条件性跳过部分组件；NoRos 机型（V7，无咖啡机器人版本）仅初始化前面板嵌入式/前面板温湿度/前面板扫码器 + PLC

#### jobs start — 异步任务执行器

通用异步任务执行框架，从本地 SQLite 数据库轮询待执行任务，由 5 个 Worker 并行处理。

- 工作机制：
  1. 轮询 SQLite 中 job_store 表，查找到期且未完成的任务（状态为"初始化"或"执行中"）
  2. 取出任务后反序列化任务列表，按顺序逐个执行子任务（支持断点续跑，记录已完成的子任务偏移量）
  3. 每个子任务有 30 秒执行超时
  4. 执行失败的任务会自动重试，采用指数退避策略（2 的 n 次方秒，最短 30 秒），最多重试 3 次
  5. 3 次重试后仍失败则标记为"失败"状态
- 任务来源：其他服务（如 brain、worker）通过 Job.ExecuteBackend() 将任务序列化后写入 SQLite，jobs 进程负责取出并执行
- 当前状态：框架已搭建完成，internal/tasks/init.go 中暂无注册的生产任务类型，属于预留的扩展能力

#### check property start — 设备自检

设备启动时执行的自检流程：

1. 读取本机 MAC 地址，通过 CRC32 生成唯一 UUID
2. 调用云端自检接口，传入机型和 UUID，获取设备配置（设备名、机器ID、机器码、机型、产品密钥）
3. 将配置信息写入本地 SQLite 持久化存储
4. 后续服务启动时从 SQLite 读取配置，无需重复请求云端

#### 其他子命令

- version — 输出版本信息（构建时间、Go 版本、Git commit 等）
  此外，服务启动时会在端口 6060 开启 pprof 性能调试服务，同时初始化 Redis 延迟队列和 RedLock 分布式锁

1. iot/script/script_scan/main.go — 扫码枪服务
   扫码枪专用服务，独立运行，处理条码/二维码扫描。

#### 工作模式

- scan code start — 正式扫码模式：
  1. 初始化硬件串口，连接扫码枪
  2. 循环读取串口数据，获取扫码内容
  3. 调用 brain 后端接口校验扫码结果（取餐码验证、订单查询）
  4. 根据返回结果播放对应语音（成功/失败/重复取餐等）
  5. 将有效扫码结果写入 Redis 队列，供其他服务消费
- debug — 调试模式：
  - 通过 MQTT 模拟扫码事件（每 2 秒发送一条），用于测试不同扫码结果的处理逻辑
  - 模拟场景：扫码中、扫码成功、重复扫码取餐、订单不存在、订单状态错误

#### 应用场景

顾客取餐扫码、配料入库扫码、设备维护扫码  
6. iot/script/property/main.go — 设备属性上传
设备属性批量上传服务，根据机型选择对应的属性采集策略。

#### 工作流程

1. 读取当前机型配置，实例化对应的属性采集器
2. 持续采集各硬件组件属性数据
3. 批量上报到云端

#### 采集内容

各组件在线状态、传感器读数、固件版本、错误码、剩余容量等

#### 应用场景

运维人员通过后台查看设备实时状态、异常告警触发  
7. iot/script/cron/main.go — IoT 设备端定时任务
IoT 设备端定时任务，工控机本地的周期性维护任务。

- 定时任务：每天 11:00 清理摄像头录像文件夹（/home/hooloo-ftp/share），遍历所有摄像头子目录，按日期排序文件夹，保留最新 15 天的录像，自动删除更早的文件
- 注意：根 Makefile 的 build_cron 目标实际编译的是此文件，而非根目录的 cron/main.go

## 五、iot/goros/ — ROS 节点服务（机器人通信）

以下服务作为 ROS (Robot Operating System) 节点运行在工控机上，与本地 ROS Master 通信。每个节点控制一种特定的硬件设备，通过 ROS Action/Service/Topic 协议交互。

1. iot/goros/sensors_plc_pub/main.go — PLC 取货
   PLC 控制器 Action 服务器。PLC（可编程逻辑控制器）是机器人的"手"，负责所有物理取放动作。

- 业务流程：接收取货指令（包含仓位编号列表）→ 控制 PLC 从指定仓位取出杯子/原料 → 返回成功/失败
- 仓位系统：机器内部有多层货架，每个仓位存放不同的杯子或原料，PLC 根据指令从对应位置取物
- 参数校验：仓位编号必须大于 0，Extra 列表不能为空
- 当前状态：PLC 调用代码被注释掉（可能由 iot/script 直接通过串口控制 PLC，不走 ROS）

1. iot/goros/tag_machine_action_server/main.go — 标签机
   标签打印机 Action 服务器。每杯饮品制作完成后，自动打印标签贴在杯子上。

- 标签内容：品牌名（如"小葫芦"）、二维码内容（订单编号等）、制作时间
- 打印参数：每项内容包括文字内容、X/Y 坐标、字号 — 精确控制标签上每个元素的位置
- 反馈机制：循环查询打印进度百分比，通过 ROS Feedback 通道实时回报
- 当前状态：代码结构完整，但硬件串口通信部分被注释掉（可能由 iot/script 直接控制）

1. iot/goros/scan_server/main.go — 扫码枪
   扫码枪 ROS 节点。通过 ROS Topic 订阅/发布机制触发扫码和上报结果。

- 触发方式：订阅 TopicScanSub，当收到触发指令时启动扫码枪
- 结果上报：扫码完成后发布到 TopicPropertyUploadPub（统一属性上报通道），包含组件代码、属性代码、值和状态
- 应用场景：配料入库时扫描条码验证、设备维护时扫描部件编号

1. iot/goros/robotic_arm_action_server/main.go — 机械臂
   机械臂 Action 服务器。控制多轴机械臂执行抓取、搬运、放置等动作。

- 动作类型丰富：包括原点归位、取冷杯、取常温杯、取杯盖、冰水工位、液体墙工位、果酱工位、制冰机工位、咖啡机工位（取/放）、压盖、放置取餐区、中转区（取/放）、激光定位、弃杯等 40+ 种动作
- 安全机制：每种动作都有对应的安全点（Safety），确保机械臂在异常时回到安全位置
- 当前状态：整个 main() 被注释掉（可能由 iot/script 直接通过串口控制，或由 PLC 替代）

1. iot/goros/electric_relay_srv_server/main.go — 继电器
   继电器 ROS Service 服务器。继电器用于控制需要开关操作的设备（柜门锁、灯光、风扇等）。

- 控制方式：指定继电器编号 + 持续时间 → 打开继电器 → 等待指定时长 → 关闭继电器
- 操作类型：开（1）、关（2）、查询状态（3）
- 应用场景：制作完成后自动开门、灯光指示、散热风扇定时开关
- 当前状态：整个 main() 被注释掉（可能由 iot/script 直接通过 GPIO 控制）

1. iot/goros/coffee_action_server/main.go — 咖啡机
   咖啡机 Action 服务器。驱动商用咖啡机完成咖啡萃取和出品。

- 制作流程：接收产品代码 → 驱动咖啡机执行对应配方 → 循环查询制作状态 → 反馈进度 → 完成返回成功
- 支持的指令：获取版本、获取机器状态、制作产品（DoProduct）、冲洗、停止、清洗、奶管冲洗、ETC 校准、获取计数器、获取序列号等
- 支持反馈：通过 Feedback 通道实时回报制作进度（0%~100%）
- 当前状态：整个 main() 被注释掉（可能由 iot/script 直接通过串口/TCP 控制咖啡机）

# 业务流程

> 用「顾客在机器点一杯拿铁并完成制作」这条主线，把 13 个模块的所有表串起来。每个表都标注在哪一步、读还是写、代码在哪。所有路径均来自源码核对。

## 一句话主线

```plaintext
顾客点单 → 校验店铺/商品/SKU/优惠券 → 落 SaaS 订单 → 支付回调扣库存/同步 brain
→ 查制作指令模板 → 生成订单指令 → 下发机器 → 机器执行反馈 → 扣机器余量 → 记仓库消耗
→ 余量触底预警/告警 → 飞书通知 → 严重时闭店 → 制作完成取餐
```

⚠ 异步制作流水线是 5 级队列串联（worker/cmd/hub_worker.go:245startMakeGoroutines），不是单步：① dealOrder→SplitOrderToCommandWorker(生成 order_risc + order_detail_space) → ② checkOrderDetailCoffeeSpace(咖啡机仓位检测，已分配的 space 在此校验) → ③ generateRisc→MakeSendOrderRisc(拆分生成 ros_risc_flow) → ④ sendRiscToTopic(MQTT 下发) → ⑤ dealAmqpRiscResult→DealAmqpOrderRisc(AMQP 反馈) → ⑥ dealPickResult(取餐完成)。每级之间靠 Redis List 队列解耦，失败回推重试。

## 关键 ID 与跨库同步

### 贯穿全流程的三个关键 ID

这三个 ID 是串联所有表的"线"：

1. shop_id — 从下单到闭店，所有 SaaS 表都挂在店铺下。SaaS 库的绝对中枢。
2. machine_id + ROS 编码对（com_ros_code/pro_ros_code）— 串联机型模板→机器实例→制作指令→余量监控→仓库补料。
3. order_id + detail_id — 从订单到指令到执行流水到物料消耗，跨 Master/SaaS/DAO 三层传递。

### 两个跨库同步点

- SaaS order ↔ Master order：支付后 SyncBrainOrderStatus 同步状态。SaaS 是交易主表，Master 是制作镜像。
- material（SaaS）↔ 组件/RISC（Master）：material_id 在 SaaS 库定义，被 Master 库的 model_component_property/order_risc/ros_risc_flow 跨库引用。

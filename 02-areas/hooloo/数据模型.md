## 概述

本项目使用 MySQL 数据库，ORM 层同时使用了 GORM。数据模型分布在两个主要包中：

- internal/model/ — Brain 侧核心模型（机器管理、指令控制、订单执行、飞书通知等）
- internal/model/saas/ — SaaS 平台侧模型（商品、店铺、优惠券、用户、权限等）
两套模型通过 MasterDB（Brain 库）和 SaasDB（SaaS 库）两个数据库连接分别访问。

订单状态机：

```plaintext
0 创建未校验 → 1 未支付 → 2 已支付 → 3 制作中 → 4 制作完成未取 → 5 待取超时
→ 6 正常完成
→ 7 未支付取消 | 8 支付后制作前取消 | 9 制作中取消 | 10 制作完成取消
→ 11 已退款 | 12 部分退款
→ 50 其他人工干预
```

## 四、数据流向说明

### 4.1 商品上架流程

```plaintext
1. 定义原料 → material
2. 定义规格 + 规格选项 → spec → spec_rule
3. 关联选项与原料 → spec_rule_material
4. 创建商品（含规格JSON） → goods
5. 生成SKU（规格选项组合） → goods_sku
6. 配置机型指令 → sku_model_risc（SKU + 机型 → 指令模板）
7. 店铺选品 → shop_goods + shop_goods_sku
8. 生成机器指令 → sku_machine_risc（店铺+机器维度）
```

### 4.2 订单执行流程

```plaintext
1. 用户下单 → order + order_detail（SaaS库）
2. Brain接收订单 → 同步到Brain库 order + order_detail
3. 生成订单指令 → order_risc（基于 sku_machine_risc）
4. 发送ROS指令 → ros_risc_flow（实际执行流程）
5. ROS反馈 → 更新 ros_risc_flow.state / order_risc.state
6. 制作完成 → 更新 order.state
7. 用户取餐 → 更新 order.state = 6（正常完成）
```

### 4.3 库存监控流程

```plaintext
1. 配置预警 → machine_warn_alarm（设置预警/报警值）
2. ROS上报 → machine_component_property.value 更新
3. 触发预警 → warn_alarm_log 记录
4. 飞书通知 → lark_message → lark_chat → lark_chat_user
5. 补料操作 → material_monitor + material_monitor_op 记录
```

# hooloo 数据模型 ER 图

仓库的 GORM 表结构关系图，按业务模块拆分。表结构分布在两个位置：

- internal/model/ — 原始手写模型层（brain 主库 + SaaS + SQLite）
- internal/dao/model/ — gen 生成的 DAO 模型层（仓库货品/金蝶/运营事件/绩效 等新业务）
两层共存：DAO 层的 shop/order/warehouse/event/baofoo_pay_order 等与 internal/model/ 重叠（gen 查询与手写模型并存），DAO 层独有的新表见模块 12、13。

## 三件事先知道

1. 三个数据库

| 数据库   | 变量            | 存什么                                                               |
| -------- | --------------- | -------------------------------------------------------------------- |
| MasterDB | drives.MasterDB | brain 主库：订单制作、机器指令(RISC)、机器组件、飞书消息             |
| SaasDB   | drives.SaasDB   | SaaS：店铺、商品、优惠券、顾客、支付、仓库货品、金蝶、运营事件、绩效 |
| SqliteDB | drives.SqliteDB | 设备端本地：工厂测试、POS 支付缓存                                   |

1. 没有数据库外键约束。 全仓库零 gorm:"foreignKey:..."，所有关系靠 XxxID 命名 + 手写 Joins()。图里画的"外键"都是逻辑外键。
2. 大量跨库引用。Material 表在 SaaS 库，却被 Master 库的组件/指令表引用；order 在 Master 库，却引用 SaaS 库的 shop/machine。跨库处图上已标注。

5. RISC 制作指令流 — MasterDB
全部 MasterDB，引用的 sku/shop/machine/material 全跨库。四级流转： SkuModelRisc(机型模板) → SkuMachineRisc(店铺实例化) → OrderRisc(订单行拷贝) → RosRiscFlow(ROS 执行流水)。RosRiscFlow只有 risc_code 无 risc_id。

CouponCategory→Coupon；CouponRule(CSV 适用范围)；CouponUser(领券用券，CustId 是 open_id)；CouponPackage/hooloo_activities/home_activity_users。

SysUser --< SysUserRole >-- SysRole --< SysRoleMenu >-- SysMenu；SysUserShop 桥接后台用户到业务 shop。  
8. 运营平台权限 · 日志 — SaasDB
OperationUser --< OperationUserRole >-- OperationRole --< OperationRoleMenu >-- OperationMenu。数据范围用 JSON 反范式存。下挂 OperationOpLog/OperationShopBI/OperationUploadRecord/LarkShopWebhook。  
9. 飞书消息 — MasterDB
LarkChat→LarkChatUser/LarkMessage→LarkMessageLog；LarkCardTemplate 用 template_id 字符串关联。OperatorUserID(string) 与 OperationUser.ID(uint) 类型不匹配，无强关联。

Event/ShopStatusLog(SaasDB)；Carousel→CarouselUserLog(MasterDB)；TrackEvent/Posters/PosterActivities/UserTags/MediaAssets/SpaceCapsuleMedia(SaasDB)。

## 补充说明

- internal/dao/model/ 与 internal/model/ 重叠表：shop/order/order_refund/warehouse/warehouse_shop/event/baofoo_pay_order 在两层都有定义。DAO 层是 gen 生成、用于类型安全查询；手写层用 drives.XxxDB.Table() 直接操作。两者操作同一张物理表。
- DAO 层 MasterDB 侧（internal/dao/model/model.go + internal/dao/query/）：order、multi_glasses_guide（多杯引导已读标记），以及 user/passport（示例/遗留表）。
- Customer 双主键：CustomerId int64（数字 PK）与 ID string（open_id）。多数表用 open_id 字符串引用它。
- 软删除不统一：业务表多用 IsDeleted int(0/1)；RuoYi 系用 DelFlag string；BI/Operation 嵌 gorm.Model；少数用 DeletedAt \*time.Time。
- CSV/JSON 伪外键：coupon_rule.shop_ids、image.shop_ids、carousel.shop_ids 等需 split 后才能匹配。
- 未建模：kds.go（仅响应 DTO）；brain/、iot/ 目录（无 TableName() 实体）；internal/server/、internal/service/ 里的结构体都是查询 DTO 或 scan 结果，非表。
